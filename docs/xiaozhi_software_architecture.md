# 小智 ESP32 软件整体架构调研报告

## 一、项目概述

`xiaozhi-esp32` 是一套运行在 ESP32 系列芯片上的小智 AI 终端固件。它面向带麦克风、扬声器、屏幕、网络模块、按键、触摸、摄像头或其他外设的嵌入式设备，核心目标是把端侧硬件能力与云端 AI 服务连接起来，实现语音唤醒、实时语音对话、语音播放、屏幕交互、设备控制、MCP 工具调用、设备激活和 OTA 升级。

从项目定位看，它不是单一板卡的应用程序，而是一套可复用的嵌入式 AI 设备软件框架。项目通过统一的应用状态机、音频服务、通信协议接口和板级硬件抽象，将大量不同形态的 ESP32 设备纳入同一套运行时架构中。

当前仓库的核心代码位于 `main/` 目录，主要包含：

| 目录或文件 | 职责 |
| --- | --- |
| `main/application.*` | 应用主控、事件循环、状态机、对话流程 |
| `main/audio/` | 音频采集、播放、唤醒词、AFE、Opus 编解码 |
| `main/protocols/` | MQTT+UDP、WebSocket 等云端通信协议 |
| `main/boards/` | 多板卡硬件适配层 |
| `main/display/` | LCD、OLED、LVGL、表情显示 |
| `main/led/` | 单灯、GPIO 灯、灯带等 LED 抽象 |
| `main/mcp_server.*` | 设备端 MCP 工具服务 |
| `main/ota.*` | 版本检查、激活、固件升级和协议配置 |
| `main/settings.*` | NVS 持久化配置封装 |
| `main/assets/` | 多语言文本、提示音和公共资源 |

## 二、整体架构

小智 ESP32 固件可以分为六个主要层次：

```text
云端 AI 服务
    |
通信协议层
    |
应用控制层
    |
音频与设备服务层
    |
板级硬件适配层
    |
ESP-IDF 与芯片外设驱动
```

各层职责如下：

| 层次 | 主要职责 |
| --- | --- |
| 云端 AI 服务 | 提供 ASR、LLM、TTS、设备激活、OTA 配置、MCP 客户端等能力 |
| 通信协议层 | 建立 MQTT+UDP 或 WebSocket 通道，传输控制消息和 Opus 音频 |
| 应用控制层 | 管理设备状态、唤醒、监听、说话、打断、错误处理和 UI 联动 |
| 音频与设备服务层 | 负责音频采集、唤醒词、AFE、AEC、VAD、Opus 编解码、MCP 工具 |
| 板级硬件适配层 | 适配不同屏幕、音频芯片、网络模块、电源、按键、触摸、摄像头 |
| ESP-IDF 与驱动层 | 提供 FreeRTOS、NVS、网络、I2S、LCD、摄像头、语音前端等基础能力 |

整体设计的关键点是：应用层不直接绑定具体硬件。应用只依赖 `Board`、`AudioService`、`Protocol`、`Display` 等抽象接口，具体板卡使用哪个音频芯片、哪组 I2S 引脚、哪种 LCD 驱动、哪种联网方式，都由板级适配层和编译配置决定。

## 三、核心模块说明

### 1. 应用主控模块

应用主控模块由 `Application` 类承担，是整个固件的调度中心。

它主要负责：

- 初始化显示界面、音频服务、网络和 MCP 工具。
- 监听板级网络事件并更新设备状态。
- 检查 OTA、激活状态和云端协议配置。
- 创建 MQTT 或 WebSocket 协议实例。
- 管理唤醒、监听、说话、打断和停止监听等对话流程。
- 接收云端 JSON 消息，驱动屏幕、表情、聊天内容和 MCP 工具调用。
- 统一处理错误、重启、升级和电源性能模式。

设备运行时围绕状态机工作。典型状态包括：

| 状态 | 含义 |
| --- | --- |
| `kDeviceStateStarting` | 设备启动中 |
| `kDeviceStateWifiConfiguring` | Wi-Fi 配网中 |
| `kDeviceStateIdle` | 空闲待唤醒 |
| `kDeviceStateConnecting` | 正在打开音频通道 |
| `kDeviceStateListening` | 正在监听用户语音 |
| `kDeviceStateSpeaking` | 正在播放云端回复 |
| `kDeviceStateUpgrading` | 固件或资源升级中 |
| `kDeviceStateActivating` | 设备激活中 |
| `kDeviceStateAudioTesting` | 音频测试中 |
| `kDeviceStateFatalError` | 致命错误 |

`Application` 使用 FreeRTOS EventGroup 聚合来自音频、网络、按键、唤醒词、定时器和协议层的事件，然后在主循环中统一处理。这种方式避免多个任务直接修改状态和 UI，有利于维持状态一致性。

### 2. 音频服务模块

音频服务由 `AudioService` 负责，是语音交互能力的核心。它把音频输入、语音前处理、唤醒词检测、Opus 编码、网络发送、Opus 解码和扬声器播放组织成一条完整链路。

音频上行链路：

```text
麦克风 / I2S
  -> AudioCodec 读取 PCM
  -> 重采样到处理采样率
  -> WakeWord 或 AudioProcessor
  -> AFE / VAD / AEC / 降噪
  -> Opus 编码
  -> audio_send_queue
  -> Protocol 发送到云端
```

音频下行链路：

```text
云端返回 Opus 音频
  -> Protocol 接收音频包
  -> audio_decode_queue
  -> Opus 解码
  -> 重采样到播放采样率
  -> audio_playback_queue
  -> AudioCodec 写入扬声器
```

音频服务内部主要包含以下角色：

| 模块 | 职责 |
| --- | --- |
| `AudioCodec` | 屏蔽 ES8311、ES8374、ES8388、ES8389、无 Codec I2S 等硬件差异 |
| `AudioProcessor` | 处理麦克风输入，可使用 AFE 实现 AEC、VAD、降噪等 |
| `WakeWord` | 实现唤醒词检测，支持 AFE、自定义模型和 ESP 唤醒词 |
| `OpusEncoderWrapper` | 将 PCM 编码为低延迟 Opus 音频包 |
| `OpusDecoderWrapper` | 将云端 Opus 音频解码为 PCM |
| `OpusResampler` | 在硬件采样率、处理采样率和播放采样率之间转换 |

对于带扬声器的小智设备，AEC 和实时打断体验非常重要。设备播放 TTS 时仍可能需要继续监听用户输入，因此音频链路需要同时考虑采集、播放、回声消除和状态切换。

### 3. 通信协议模块

通信协议层由 `Protocol` 抽象类统一管理。上层应用不直接关心底层是 MQTT、UDP 还是 WebSocket，只通过统一接口完成以下动作：

- 打开音频通道。
- 关闭音频通道。
- 发送 Opus 音频包。
- 发送开始监听、停止监听、打断说话、唤醒词检测等控制消息。
- 接收云端音频包。
- 接收云端 JSON 消息。
- 处理网络错误和连接关闭。

当前项目支持两类主要通信方式：

| 协议 | 特点 |
| --- | --- |
| MQTT + UDP | MQTT 传输控制消息，UDP 传输加密后的 Opus 音频，控制和音频分离 |
| WebSocket | 单连接同时传输 JSON 控制消息和二进制音频帧，结构更简单，便于调试 |

协议选择通常来自 OTA 配置。设备启动后先向服务器请求版本和配置，服务器返回可用的 MQTT 或 WebSocket 参数，应用再根据配置创建具体协议实例。若没有明确配置，当前逻辑会回退到 MQTT。

云端 JSON 消息主要包括：

| 类型 | 作用 |
| --- | --- |
| `tts` | 通知 TTS 开始、结束或句子开始，驱动播放状态和字幕 |
| `stt` | 返回用户语音识别文本 |
| `llm` | 返回模型情绪，用于更新表情 |
| `mcp` | 承载 MCP JSON-RPC 消息 |
| `system` | 执行系统命令，如重启 |
| `alert` | 显示提醒并播放提示音 |
| `custom` | 可选的自定义业务消息 |

### 4. 板级硬件适配模块

板级适配层是本项目支持大量硬件形态的关键。基础接口是 `Board`，不同设备通过继承 `Board` 或其派生类暴露统一能力。

常见基类包括：

| 基类 | 适用场景 |
| --- | --- |
| `Board` | 所有板卡的基础抽象 |
| `WifiBoard` | Wi-Fi 联网设备 |
| `Ml307Board` | 使用 ML307 蜂窝网络模块的设备 |
| `Nt26Board` | 使用 NT26 类网络模块的设备 |
| `DualNetworkBoard` | 支持 Wi-Fi 与蜂窝网络切换的设备 |
| `RndisBoard` | 使用 USB RNDIS 网络的设备 |

板级适配通常负责：

- 创建并返回 `AudioCodec`。
- 创建并返回 `Display`。
- 初始化 LCD、OLED、触摸、背光和 LED。
- 初始化按键、旋钮、触摸传感器等输入设备。
- 初始化 Wi-Fi、蜂窝网络或 USB 网络。
- 提供电池、电源、省电和充电状态。
- 注册板卡专属 MCP 工具。
- 注册摄像头、灯光、电机或其他外设能力。

每个板卡目录通常包含 `config.h`、`config.json` 和一个或多个 `.cc` 实现文件。板卡类通过 `DECLARE_BOARD(...)` 注册为当前固件的板级单例。

项目通过 `Kconfig.projbuild` 选择 `Board Type`，再由 `main/CMakeLists.txt` 根据 `CONFIG_BOARD_TYPE_*` 把对应板卡源码加入编译。这使同一套应用逻辑可以面向不同硬件构建不同固件。

### 5. 显示与交互模块

显示层由 `Display` 抽象类统一接口，上层应用只调用状态、通知、表情和聊天消息接口，不关心具体屏幕类型。

主要显示实现包括：

| 类型 | 作用 |
| --- | --- |
| `LvglDisplay` | 基于 LVGL 的通用显示基类 |
| `LcdDisplay` | 面向 SPI、RGB、MIPI LCD 的显示实现 |
| `OledDisplay` | 面向小尺寸 OLED 屏幕 |
| `EmoteDisplay` | 面向表情动画资源的显示实现 |

显示内容主要包括：

- 设备状态，例如待机、连接中、监听中、说话中、升级中。
- 用户和助手聊天文本。
- LLM 情绪对应的表情。
- Wi-Fi、蜂窝网络、电量、时间等状态栏信息。
- 配网、激活、错误、音量、亮度等临时通知。

对于没有屏幕或只有简单 LED 的设备，板级层也可以返回简化显示或仅通过 LED 表示状态。

### 6. OTA、激活与配置模块

OTA 模块不仅负责固件升级，还承担设备运行配置下发的职责。

设备启动后会向服务器请求版本信息和配置，服务器可以返回：

- 是否存在新固件。
- 固件下载地址和目标版本。
- MQTT 连接配置。
- WebSocket 连接配置。
- 设备激活码和激活提示。
- 服务器时间。
- 资源版本或资源下载地址。

设备如果未激活，会显示激活码并等待云端确认。激活完成后，应用继续初始化协议并进入可对话状态。固件升级时，应用会关闭音频通道、停止音频服务、显示升级进度，并在成功后重启。

### 7. MCP 工具模块

MCP 模块让云端模型不仅可以和用户聊天，还可以发现并调用设备本地能力。项目中的 MCP 消息被封装在底层 MQTT 或 WebSocket 的 `type: "mcp"` 消息中，内部遵循 JSON-RPC 2.0 风格。

设备端 MCP 服务主要支持：

- `initialize`：初始化 MCP 会话。
- `tools/list`：列出设备可用工具。
- `tools/call`：调用指定设备工具。
- 设备主动发送 MCP notification。

通用工具能力包括：

- 获取设备状态。
- 调整扬声器音量。
- 调整屏幕亮度。
- 设置屏幕主题。
- 调用摄像头拍照。

板卡也可以注册专属工具，例如灯控、按住说话、传感器读取、电机控制、触摸状态读取等。这样小智设备就不只是语音终端，而是一个可被 AI 感知和控制的智能硬件节点。

### 8. 资源与本地化模块

项目内置多语言资源，包括：

- `language.json`：不同语言的界面文本和提示文案。
- `.ogg` 提示音：欢迎、配网、激活、错误、升级、数字播报等。
- 公共音效：振动、成功、弹窗、低电量、警告等。
- 字体、图标和表情资源。

构建系统会根据语言配置选择对应资源，并在缺失时回退到英文资源。对于带复杂屏幕的设备，还可以构建 `assets.bin` 并烧录到独立资源分区。

## 四、系统运行流程

### 1. 启动流程

设备上电后，软件从 `app_main` 进入主程序，完成基础系统初始化后进入应用主控模块。

```text
系统上电
  -> 初始化 NVS
  -> 获取 Application 单例
  -> 初始化显示 UI
  -> 初始化 AudioService
  -> 注册音频、状态、网络回调
  -> 注册 MCP 工具
  -> 启动网络
  -> 检查 OTA / 激活 / 协议配置
  -> 初始化 MQTT 或 WebSocket 协议
  -> 进入空闲待唤醒状态
```

### 2. 唤醒与对话流程

设备空闲时通常保持唤醒词检测。用户可以通过唤醒词、按键、触摸或其他板级输入触发一次对话。

```text
空闲状态
  -> 唤醒词或用户输入触发
  -> 打开云端音频通道
  -> 进入监听状态
  -> 上传用户语音
  -> 云端返回 STT 文本、LLM 情绪和 TTS 音频
  -> 设备显示文字和表情
  -> 播放 TTS 回复
  -> 回到监听或空闲状态
```

对话是否回到空闲状态，取决于监听模式。手动停止模式通常在 TTS 结束后回到空闲；实时模式或自动停止模式可能继续保持监听。

### 3. 打断流程

当设备正在播放云端回复时，如果用户再次唤醒、按键或触发打断，应用会通知云端中断当前 TTS，并清理本地播放队列。

```text
设备说话中
  -> 用户触发打断
  -> 发送 abort 消息
  -> 停止当前播放
  -> 根据模式进入监听或空闲状态
```

这种机制让设备具备更自然的对话体验，用户不必等待设备完整说完。

### 4. 网络与协议恢复流程

网络连接由板级网络模块负责。网络事件通过 `Board` 回调通知 `Application`。当网络断开、协议错误或音频通道关闭时，应用会更新状态和 UI，并在必要时关闭音频通道或重新进入空闲状态。

```text
网络事件
  -> Board 统一为 NetworkEvent
  -> Application 接收事件
  -> 更新显示和状态
  -> 必要时关闭协议音频通道
  -> 等待网络恢复或重新连接
```

### 5. OTA 升级流程

```text
检查版本
  -> 云端返回新版本
  -> 显示升级提示
  -> 关闭音频通道
  -> 停止音频服务
  -> 下载并写入固件
  -> 升级成功后重启
```

资源升级流程与固件升级类似，但目标是资源分区，例如提示音、表情和字体资源。

## 五、主要数据流

### 1. 音频数据流

```text
用户声音
  -> 麦克风
  -> AudioCodec
  -> AudioService
  -> AFE / WakeWord / Opus
  -> Protocol
  -> 云端 AI 服务
```

```text
云端 TTS
  -> Protocol
  -> AudioService
  -> Opus 解码
  -> AudioCodec
  -> 扬声器
```

### 2. 控制消息流

```text
云端 JSON
  -> Protocol
  -> Application
  -> 状态机 / Display / AudioService / McpServer
```

```text
设备本地事件
  -> Board 或 AudioService
  -> Application EventGroup
  -> 主事件循环
  -> Protocol / Display / AudioService
```

### 3. MCP 工具调用流

```text
云端模型
  -> MCP Client
  -> type: "mcp" 消息
  -> Protocol
  -> McpServer
  -> 本地工具回调
  -> 设备硬件或状态
  -> MCP 调用结果返回云端
```

## 六、构建与配置体系

项目通过 ESP-IDF 构建系统组织编译。

### 1. 板卡选择

`main/Kconfig.projbuild` 提供 `Board Type` 选项。用户在 `menuconfig` 中选择具体板卡后，会生成对应的 `CONFIG_BOARD_TYPE_*` 宏。

`main/CMakeLists.txt` 根据该宏设置：

- `BOARD_TYPE`
- 板卡源码目录
- 字体和图标资源
- 默认表情资源
- 额外资源文件
- 条件编译的摄像头、视频、唤醒词和音频处理模块

### 2. 组件依赖

`main/idf_component.yml` 管理外部组件依赖，主要包括：

- LCD、触摸和 IO 扩展芯片驱动。
- LVGL 和 ESP LVGL Port。
- ESP-SR 语音识别和唤醒词组件。
- ESP Audio Codec 和 Audio Effects。
- ML307、UART 网络模块。
- 摄像头、视频和图像处理组件。
- 字体、表情、mmap assets 等资源组件。

### 3. 分区与资源

项目包含 `partitions/v1` 和 `partitions/v2` 分区表。v2 与 v1 不兼容，资源通常可以独立烧录到 `assets` 分区。构建系统支持：

- 不烧录资源。
- 烧录默认生成资源。
- 烧录自定义 `assets.bin`。
- 生成表情资源。

## 七、扩展开发方式

### 1. 新增板卡

新增板卡通常需要：

1. 在 `main/boards/<board-name>/` 下创建板卡目录。
2. 添加 `config.h` 和 `config.json`。
3. 编写板卡 `.cc` 文件，继承 `WifiBoard`、`Ml307Board`、`DualNetworkBoard` 或 `Board`。
4. 实现音频、显示、网络、按键、LED、电源等初始化。
5. 使用 `DECLARE_BOARD(...)` 注册板卡。
6. 在 `Kconfig.projbuild` 和 `CMakeLists.txt` 中加入板卡选项。

### 2. 新增 MCP 工具

新增设备能力时，可以通过 `McpServer::AddTool` 注册工具。工具应明确描述：

- 工具名称。
- 工具用途。
- 参数列表。
- 参数类型和是否必填。
- 返回值含义。

工具命名建议使用 `self.xxx.yyy` 风格，表示这是设备自身能力。

### 3. 新增协议

新增通信方式时，应继承 `Protocol`，实现：

- 打开和关闭音频通道。
- 发送文本消息。
- 发送音频包。
- 解析服务端 hello。
- 触发音频、JSON、错误和连接状态回调。

然后在 `Application::InitializeProtocol()` 中加入选择逻辑。

### 4. 新增音频处理能力

音频处理能力可以通过扩展 `AudioProcessor`、`WakeWord` 或 `AudioCodec` 接入。开发时需要重点关注：

- 输入输出采样率。
- 单声道、双声道或参考通道格式。
- AEC 配置。
- Opus 帧长。
- FreeRTOS 任务栈和优先级。
- 队列长度和实时性。

## 八、架构优势

### 1. 模块边界清晰

应用控制、音频处理、通信协议、板级硬件、显示、MCP、OTA 和资源管理都有明确边界。每个模块职责相对独立，便于维护和定位问题。

### 2. 多硬件复用能力强

大量板卡通过 `Board` 抽象接入，同一套 `Application`、`AudioService` 和 `Protocol` 可以复用于不同产品形态。

### 3. 语音链路完整

项目覆盖从麦克风采集、唤醒词、语音前处理、Opus 编码、云端传输，到 Opus 解码、播放和打断的完整实时语音链路。

### 4. 云端配置灵活

服务器可以通过 OTA 接口下发协议配置、激活信息和升级信息，使设备端不需要固定写死所有服务参数。

### 5. 支持 AI 控制本地硬件

MCP 工具机制让云端模型能够发现和调用设备本地能力，使设备从“语音聊天终端”升级为“可被 AI 控制的智能硬件终端”。

### 6. 本地化和产品化基础较完整

项目内置多语言文本、提示音、表情、字体、分区和资源烧录机制，适合面向不同屏幕尺寸、语言环境和硬件规格做产品化适配。

## 九、当前架构的关注点

### 1. 状态机复杂度较高

唤醒、监听、说话、打断、激活、升级、配网、音频测试等状态都集中在 `Application` 中。后续新增功能时，应优先通过清晰事件和状态迁移接入，避免在板级代码或异步任务中直接修改全局状态。

### 2. 音频链路受配置影响明显

AFE、AEC、唤醒词、采样率、Opus 帧长、输入通道和参考通道都可能影响最终语音效果。调试语音问题时，需要同时检查编译配置、板级 `config.h`、音频芯片初始化和云端协议参数。

### 3. 多板卡带来维护成本

板卡数量较多，新增公共接口时需要考虑不同板卡是否实现完整。建议保持 `Board` 抽象稳定，把差异尽量限制在板级目录内。

### 4. MCP 工具需要规范化

工具名称、描述、参数和返回值会直接影响云端模型调用质量。新增工具时应保持命名清晰、参数简单、返回可解释，并避免让模型误调用危险操作。

### 5. 资源和分区需要保持版本一致

固件、资源分区、字体、表情、语言资源和 OTA 版本之间存在配套关系。升级或改分区时，需要特别注意 v1/v2 分区兼容性和资源烧录策略。

## 十、总结

小智 ESP32 软件架构可以概括为：

> 以 `Application` 为主控状态机，以 `AudioService` 为实时语音处理核心，以 `Protocol` 连接云端 AI 服务，以 `Board` 屏蔽多硬件差异，并通过 OTA、MCP、Display 和 Assets 构成完整的智能硬件产品化能力。

这套架构的价值在于，它把嵌入式硬件、实时音频、云端 AI、设备控制和 OTA 运营能力整合成一个统一框架。对于具体产品开发来说，板卡适配只需要围绕硬件能力展开；对于上层功能开发来说，语音对话、MCP 工具、协议消息和显示状态已经具备较清晰的扩展路径。

后续如果要继续演进，建议重点围绕三条主线推进：

1. 保持 `Application` 状态机可理解、可验证。
2. 建立更系统的音频调试和板卡验收流程。
3. 规范 MCP 工具设计，让 AI 调用设备能力更稳定、更可控。
